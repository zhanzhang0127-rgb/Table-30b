# 开发会话日志 — 2026-04-26（社区化转向）

本次会话在原「餐厅关联 + 独立排行榜」方案基础上，基于真实使用反馈完成了一次产品级简化：

- 从「餐厅实体驱动」转为「社区真实评价驱动」
- 从「独立排行榜页」转为「社区内置排序（最新 / 最热）」
- 从「底部中间发布按钮」转为「社区顶部发布按钮」

---

## 1. 触发背景

用户在联调中反馈：

1. 发帖必须先关联餐厅，造成发布链路阻塞
2. 即使输入已录入餐厅名，也经常显示搜不到
3. 餐厅数据录入维护成本高，不符合团队当前精力与 Demo 目标
4. 移除餐厅入口后，手机端底部导航出现布局不对称

这与产品「真实评价社区」定位一致：优先保证发帖与内容流可用，而不是维护大规模商家信息库。

---

## 2. 关键决策

### 决策 A：去餐厅化（用户侧）

- 取消发帖「必须关联餐厅」
- 隐藏用户侧餐厅与独立排行榜入口
- 保留社区、AI 助手、个人三大入口

### 决策 B：排行榜并入社区

- 不再单独保留排行榜页作为用户主路径
- 在社区页顶部加入两个分类按钮：最新 / 最热
- 最热定义为：点赞数 + 评论数 综合排序（同分按发布时间降序）

### 决策 C：手机端发布入口上移

- 手机端底部菜单仅保留：社区 / AI助手 / 个人
- 发布按钮移到社区页顶部，兼顾对称性与可达性

---

## 3. 代码改动摘要

### 3.1 后端：Feed 排序能力

- 文件：server/db.ts
  - 新增 FeedSort 类型：latest | hottest
  - 扩展 getPostsForFeed(limit, offset, sort)
  - hottest 排序：COALESCE(likes, 0) + COALESCE(comments, 0) 降序，再按 createdAt 降序

- 文件：server/routers.ts
  - posts.getFeed 输入新增 sort 枚举（默认 latest）
  - 调用 db.getPostsForFeed(input.limit, input.offset, input.sort)

### 3.2 前端：发帖页去餐厅依赖

- 文件：client/src/pages/Publish.tsx
  - 删除 restaurantId 必填校验
  - 删除餐厅搜索/下拉选择器状态与 UI
  - 删除发布按钮对 restaurantId 的 disable 依赖
  - 保留语音提取结果中的「提到的餐厅」文本，仅作为内容提示，不做实体绑定

### 3.3 前端：社区页内置最新/最热

- 文件：client/src/pages/Feed.tsx
  - 新增 sort 状态与顶部切换按钮
  - 切换 sort 时重置 offset 与列表
  - 请求参数带 sort
  - 在社区页顶部新增发布按钮

### 3.4 前端：导航与路由简化

- 文件：client/src/components/ResponsiveNav.tsx
  - 底部导航改为 3 项：社区 / AI助手 / 个人
  - 删除底部中间悬浮发布按钮

- 文件：client/src/App.tsx
  - 移除用户侧路由入口：/restaurants、/rankings、/restaurant/:id、/submit-restaurant

- 文件：client/src/pages/Home.tsx
  - 首页文案从「发现排行榜」改为「最新与最热」

## 5. 当前状态

### 已达成

- 发布主路径明显简化：登录 → 社区 → 发布
- 社区信息流具备即时分类：最新 / 最热
- 手机端底部导航恢复对称且一致

### 仍保留（未物理删除）

- 仓库内仍存在部分餐厅与排行榜相关页面/后端代码（不在用户主路径中）
- 这样做是为降低一次性大删引发回归的风险

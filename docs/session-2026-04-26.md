## 2. 代码改动

### 2.1 [server/_core/glm4.ts](../server/_core/glm4.ts) — 支持 JSON 模式

**修改**：`invokeGLM4` options 新增可选字段 `responseFormat?: { type: "json_object" }`，在请求 body 中透传为 `response_format`。保持向后兼容（不传则行为不变）。

### 2.2 [server/_core/voicePostExtractor.ts](../server/_core/voicePostExtractor.ts) — 新增

单一导出函数 `extractPostFromTranscript(transcript: string): Promise<ExtractedPost>`。

- **输入**：口语化语音转写文本
- **输出**：`{ title, content, rating, restaurantNameHint, recommendedDish }`
- **System prompt 核心约束**：
  - 保留用户原话真实感，拒绝任何夸张/广告口吻（去商业化红线）
  - `rating` 未明确提及时返回 `null`，不脑补评分
  - 输出严格 JSON，调用 GLM-4 JSON 模式（temperature 0.3）
- 包含 `normalizeExtracted` 做字段类型兜底 + 长度截断（title ≤ 100，content ≤ 1000）

### 2.3 [server/_core/voiceTranscription.ts](../server/_core/voiceTranscription.ts) — 重构

原 `transcribeAudio`（URL → Whisper）拆分为：

- **新增** `callWhisper(buffer, mime, language, prompt)`：核心 Whisper API 调用逻辑（内部函数）
- **新增** `transcribeBuffer({ buffer, mimeType, language, prompt })`：接受内存 Buffer，直接调用 Whisper，**跳过存储 URL 下载**（性能优化核心）
- **保留** `transcribeAudio(options: { audioUrl })`：重构为调用 `transcribeBuffer`，保持原接口不变

### 2.4 [server/routers.ts](../server/routers.ts) — 新增 voice router

新增顶层路由 `voice`，含三个 `protectedProcedure`：

| 过程 | 说明 |
|---|---|
| `uploadAudio` | base64 dataUrl → storagePut → 返回 URL（保留备用，前端已不调用）|
| `transcribe` | URL → 下载 → Whisper（保留兼容性）|
| `transcribeDirect` | base64 dataUrl → Buffer → 直发 Whisper（**主路径，性能优化版**）|
| `extractPost` | 转写文本 → GLM-4 JSON 模式 → `ExtractedPost` |

Base64 解析修复：用 `indexOf(';base64,')` 定位而非 regex，正确处理 `audio/webm;codecs=opus` 等含 codec 参数的 MIME 类型（原始版本触发过「音频必须为 base64 data URL」报错，此次修复）。

### 2.5 [client/src/pages/Publish.tsx](../client/src/pages/Publish.tsx) — 语音录音 UI

在标题输入框上方插入虚线卡片区块，**不修改**现有手填流程：

**UI 布局**：
```

│ ✨ 用语音整理你的体验（实验功能）                  │
```

## 3. 性能迭代（第二轮，同次会话）

### 3.1 问题
用户反馈：从录音结束到表单回填耗时过长。

### 3.2 根因分析

原始管道含 **4 次串行网络跳转**：

```
Client → Server（上传 base64）
Server → Forge Storage（storagePut，第1跳）
Server ← Forge Storage（transcribeAudio 内部下载，第2跳，多余！）
Server → Whisper API（第3跳）
Server → GLM-4 API（第4跳）
```

第 2 跳完全多余：服务器刚上传完，又从 URL 把同一份数据下载回来。

### 3.3 优化方案

引入 `transcribeDirect` + `transcribeBuffer`，绕过存储层：

```
Client → Server（base64 dataUrl）
Server → Whisper API（Buffer 直发，第1跳）
Server → GLM-4 API（第2跳）
```

节省 2 次存储 API 来回（Forge 上传 + Forge 下载），预计节省 **3–5 秒**，前端 loading 状态简化为 2 态。

## 5. 数据库隔离问题确认

**现象**：本地 `localhost:3000` 发的帖子，在 `chileoma-xzewdxgs.manus.space` 上也能看到。

**根因**：本地 `.env` 的 `DATABASE_URL` 与 Manus 部署共用同一个远程 TiDB 实例。代码（功能）隔离，数据不隔离。

**决策（方案 A）**：接受现状，Demo 前手动清理测试帖。不引入新 DB 实例，避免增加维护成本。

**⚠️ 高危动作提醒**：在本地运行 `pnpm db:push` 或 `migrate.mjs` 会 ALTER TABLE 共享数据库，直接影响生产网页，当前 session 期间**未执行任何 schema 改动**。

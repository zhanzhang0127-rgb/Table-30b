## 3. 诊断与决策（不涉及代码改动）

### 3.1 "验证码发不出"问题定性

**结论**：**不在我们的代码里**。整个仓库 grep `verif|captcha|sendCode|smtp|otp|验证码` 没有任何业务逻辑，注册/登录全部委托给 Manus OAuth 外部服务。验证码由 Manus 邮件后端发送。

**应对**：
- 不可在代码层修复
- 已建议联系 Manus 客服 + 用非 XJTLU 邮箱测试隔离故障类型
- 兜底方案 = dev-login（已实现），让本地能用 owner 账号登录绕过验证码

### 3.2 CloudFront 403 问题定性

**症状**：本地 `pnpm dev` 起来后点击登录，跳转到 `manus.im/app-auth?...&redirectUri=http://localhost:3000/...` → CloudFront 返回 403

**最可能原因**：Manus OAuth 应用的回调地址白名单里没有 `http://localhost:3000/api/oauth/callback`（应用注册时填的是生产域名）

**应对**：dev-login 已绕过该路径；如要真正修，需登 Manus 控制台把 localhost 回调加进白名单

### 3.3 "换 Railway 部署"问题澄清

用户提议换 Railway → **概念混淆**：
- Railway = 托管平台（同 Vercel/Render）
- Manus OAuth = 认证服务（同 Auth0/Clerk）
- 两者正交，换托管不会修认证；新部署的 URL 同样不在 Manus 白名单里 → 仍 403

**结论**：12 天内**不应换认证提供商**。理由：
1. 任何 OAuth 切换都要 4-6h+ 调试
2. Google/Microsoft OAuth 不一定能拿到 `@xjtlu.edu.cn` 后缀邮件，反而破坏"封闭性"差异化故事
3. 演讲风险：认证改动可能引入新 bug

### 3.4 三大优先级（除部署外）

按"丢分风险 × 时间不可压缩性"排序，今天就该开工的三件事：

1. **解决/绕过登录回归** — 25 分（用户验证 10% + 原型演示 15%）的前置依赖；今天必须有结论
2. **启动用户测试招募 + 任务卡设计** — 单项最大可争取分（10%），需要日历时间做"测 → 改 → 再测"两轮 before/after
3. **种子数据填充 + 核心 demo 剧本固化** — 上一步的前置（被测试者不能面对空 feed）
